06.5 — منظومة الاختبارات

🧪 Testing Suite — 113 اختبار إنتاجي مُقوّى

اختبارات تكاملية حقيقية ضد PostgreSQL. لا mocks، لا اختبارات وهمية. كل اختبار يحمي قاعدة عمل محددة. مُعزَّزة باختبارات ضغط وتزامن وأمان.

📖 أولاً: ما هي الاختبارات ولماذا نحتاجها؟

الاختبار (Test) هو كود يتحقق أن الكود الأصلي يعمل بالشكل الصحيح.

بدل ما تفتح التطبيق وتجرب يدوياً كل مرة، الاختبارات تعمل هذا تلقائياً في ثوانٍ.

مثال مبسّط:

// أعطي المحرك: TRANSPORT_SIMPLE عادي 0km
// أتوقع: السعر = 150 + ضريبة 15 = 165 MAD
const result = await calculatePrice(input);
expect(result.totalTtc).toBe(165); // ✅ نجح!

لماذا هذا مهم؟
لأن MediTrans نظام مالي — أي خطأ في التسعير = خسارة مالية حقيقية.
الاختبارات تكتشف الأخطاء قبل أن تصل للمستخدم.

⚙️ كيف تعمل الاختبارات؟

كل اختبار يمر بـ 3 مراحل بسيطة:

1️⃣
التحضير (Arrange)
أنشئ البيانات المطلوبة (مريض، خدمة، فاتورة...)
2️⃣
التنفيذ (Act)
شغّل الدالة المطلوب اختبارها
3️⃣
التحقق (Assert)
تأكد أن النتيجة = ما تتوقعه

بعد كل اختبار، المرحلة الرابعة (التنظيف / Cleanup)
تحذف كل البيانات التي أنشأها الاختبار — حتى لا يؤثر على الاختبار التالي.

🛠️ أدوات الاختبار المستخدمة

⚡ Vitest
ما هو؟ أداة تشغيل الاختبارات — أسرع من Jest.
يدعم TypeScript مباشرة بدون إعداد إضافي.
يكتب لك: ✓ passed أو ✗ failed لكل اختبار.
🗄️ Real PostgreSQL
ما هو؟ لا نستخدم قاعدة بيانات وهمية.
كل اختبار يتصل بنفس PostgreSQL الحقيقي.
الفائدة: نفس السلوك في الاختبار = نفس السلوك في الإنتاج.
🔒 العزل (Isolation)
ما هو؟ كل اختبار مستقل بذاته.
ينشئ بياناته ← يشتغل ← ينظف بياناته.
الفائدة: لا تأثير متبادل بين الاختبارات — نتائج موثوقة دائماً.

📋 فئات الاختبارات — ماذا يختبر كل ملف؟

💰 engine.test.ts — محرك التسعير (29 اختبار)

ماذا يختبر؟ الدالة الأهم في النظام: calculatePrice(). كل سيناريو تسعير ممكن:

✅ السعر الأساسي لكل كتالوج (transport simple, ECG, suture...)
✅ رسوم المسافة (10km, 20km, 0km, بدون مسافة)
✅ المعدِّلات (NIGHT_SURCHARGE نشط، VIP غير نشط، معدِّل غير موجود)
✅ حساب الضريبة TVA = 10%
✅ التجاوز اليدوي (admin فقط، سبب ≥ 10 أحرف)
✅ حالات الخطأ (كتالوج غير موجود، قاعدة تسعير غير موجودة)
✅ حفظ اللقطة في DB (persist=true/false)
📸 snapshot.test.ts — سلامة اللقطات (6 اختبارات)

ماذا يختبر؟ أن كل خدمة لها لقطة واحدة فعّالة فقط (isCurrent=true) في أي لحظة. وأن الإصدارات تتزايد بشكل صحيح (1→2→3).

✅ أول لقطة: version=1, isCurrent=true
✅ بعد تجاوز: القديمة تصبح false، الجديدة version=2
✅ INVARIANT: دائماً لقطة واحدة فقط isCurrent=true لكل خدمة
🧾 invoice.test.ts — قواعد الفوترة (5 اختبارات)

ماذا يختبر؟ القواعد الحرجة للفوترة:

✅ CR-07: خدمة ملغاة لا يمكن فوترتها
✅ CR-08: نفس الخدمة لا تظهر في فاتورتين (unique constraint)
✅ CR-09: مجموع الفاتورة = مجموع سطور اللقطات
✅ CR-12: فاتورة بها دفعات لا يمكن إلغاؤها
💳 payment.test.ts — قيود الدفع (9 اختبارات)

✅ دفعة جزئية → الحالة تصبح partial
✅ دفعة كاملة → الحالة تصبح paid
✅ تجاوز المبلغ → مرفوض
✅ دفعة على فاتورة ≠ لا تؤثر على فاتورة أخرى (عزل)

🛡️ اختبارات التقوية (HARDENING++) — 64 اختبار إضافي

هذه اختبارات متقدمة تحمي النظام من الحالات النادرة والخطيرة — كأنك تختبر سيارة في أسوأ الظروف:

💣 اختبارات الضغط — 25 اختبار
ماذا لو أعطيناه مسافة سالبة؟
ماذا لو المسافة مليون كيلومتر؟
ماذا لو سعر التجاوز = 0 أو سالب؟
ماذا لو كود الكتالوج 500 حرف؟
ماذا لو 100 معدِّل وهمي دفعة واحدة؟
← النظام إما يرفض بوضوح أو يحسب بدون crash
⚡ اختبارات التزامن — 4 اختبارات
ماذا لو شخصان حسبا السعر في نفس اللحظة؟
ماذا لو تجاوزان حصلا بالتزامن؟
← دائماً لقطة واحدة فقط isCurrent=true
← لا تكرار في أرقام الإصدارات
← لا NaN أو Infinity في الأسعار
🔗 اختبارات التكامل — 13 اختبار
التدفق الكامل من البداية للنهاية:
إنشاء خدمة → تسعير تلقائي → لقطة محفوظة
إنشاء فاتورة → خدمتان → المبلغ صحيح
تسجيل دفعة → الحالة تتحدث تلقائياً
محاولة إلغاء فاتورة مدفوعة → مرفوض
🔐 اختبارات الأمان — 22 اختبار
حقن SQL: '; DROP TABLE -- → آمن
XSS: <script>alert()</script> → آمن
قيم null/undefined/NaN/Infinity → Zod يرفضها
أنواع خاطئة (رقم بدل نص) → Zod يرفضها
Type guards → كل الأدوار والحالات مُتحقق منها

👀 كيف تقرأ نتائج الاختبار؟

عندما تشغّل npm test، سترى شيئاً مثل هذا:

✓ tests/pricing/engine.test.ts > calculatePrice — core pricing > normal transport simple 14ms ✓ tests/pricing/engine.test.ts > calculatePrice — core pricing > urgent transport simple 3ms ✗ tests/pricing/engine.test.ts > error test > (مثال فشل وهمي) ← لو ظهر هذا Test Files 8 passed (8) Tests 113 passed (113) Duration 26s
✓ (أخضر) = الاختبار نجح — النتيجة طابقت المتوقع
✗ (أحمر) = الاختبار فشل — يجب إصلاح الكود!
ms = المدة بالميلي ثانية — كلما كان أقل كان أفضل
Test Files 8 passed = كل الملفات الـ 8 نجحت
Tests 113 passed = كل الـ 113 اختبار نجحوا

📈 كيف تقرأ تقرير التغطية (Coverage)؟

التغطية تعني: كم من كودك تمر عليه الاختبارات فعلاً؟

عندما تشغّل npm run test:coverage:

File | % Lines | المعنى ─────────────────────────────────────────────── pricing.engine.ts | 94.64% | ← ممتاز! تقريباً كل سطر مختبر pricing.utils.ts | 100.00% | ← مثالي! invoice.service.ts | 84.21% | ← جيد، بعض الأسطر غير مغطاة client.repository.ts | 0.00% | ← لا اختبارات بعد (مرحلة قادمة)
90%+ = ممتاز — الملف محمي جيداً
70-89% = جيد — بعض الحالات النادرة غير مغطاة
أقل من 50% = يحتاج اختبارات إضافية
0% = الملف لم يُختبر بعد (عادي لوحدات لم تُبنى بعد)
